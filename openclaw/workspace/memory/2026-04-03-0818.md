what is contextengine
assistant: **ContextEngine** is a pluggable interface in OpenClaw (available starting with version 2026.03.07) that gives you full control over how the agent’s memory pipeline works—specifically how context is **ingested, assembled, compacted, and cleaned up** after each turn.

Instead of relying on OpenClaw’s default compaction algorithm (which can silently drop important instructions as the conversation grows), ContextEngine lets you write custom JavaScript hooks that decide exactly what stays, what gets summarized, what gets removed, and what gets isolated for sub‑agents.

### The six lifecycle hooks

| Hook | When it runs | What you can do |
|------|--------------|-----------------|
| **bootstrap** | Agent starts | Load external data, connect to a RAG vector store, initialize caches |
| **ingest** | New context arrives (user message, tool result, etc.) | Filter, tag, prioritize incoming chunks before they hit the LLM |
| **assemble** | Right before the LLM sees the prompt | Build the final prompt using custom RAG, re‑ranking, or compression logic |
| **compact** | When the context window is full and needs to be trimmed | Choose which chunks to keep, drop, or summarize—**this is where you protect critical files like SOUL.md/IDENTITY.md** |
| **afterTurn** | After the agent responds | Log, analyze, learn, update caches, prepare for next turn |
| **prepareSubagentSpawn** | Before spawning a sub‑agent | Set up isolated memory for the sub‑agent (e.g., give it only the pieces it needs) |

### Why it matters for Step 8 (Memory Architecture)

- **Memory decay** happens because OpenClaw’s default compaction may toss out the very instructions that make your agent an execution‑first engine (the “ship first, talk second” rule in SOUL.md, your identity, user prefs, etc.).
- With ContextEngine you can write a **compact hook** that says: *“Never drop anything tagged as critical”* and then tag chunks coming from SOUL.md, IDENTITY.md, USER.md, or MEMORY.md as critical. The engine will then preserve those chunks no matter how long the session gets.
- You can also use the **assemble hook** to run a RAG query against your memory store before each response, ensuring the agent always has the relevant background info at hand.
- The **prepareSubagentSpawn** hook lets you control what a sub‑agent sees (sub‑agents only get AGENTS.md and TOOLS.md by default, so you can inject memory pieces if needed).

### How to use it (quick sketch)

1. **Check version**: `openclaw --version` – if ≥ 2026.03.07 you have ContextEngine.
2. **Create a plugin folder** (e.g., `plugins/`) and add a JavaScript file, e.g. `context-protection.js`:
   ```javascript
   module.exports = {
     compact: (chunks) => {
       // Keep any chunk marked critical (you can tag chunks during ingest/assemble)
       return chunks.filter(c => c.metadata?.critical === true);
     },
     // OPTIONAL: tag critical chunks during ingest
     ingest: (chunks) => {
       return chunks.map(c => {
         if (c.text.includes('SHIP FIRST') || c.text.includes('bbymama') || c.text.includes('USER.md')) {
           return {...c, metadata: {...c.metadata, critical: true}};
         }
         return c;
       });
     }
   };
   ```
3. **Register the hook** in your `openclaw.json` (under `contextEngine.hooks` as described in the docs) or via the CLI if OpenClaw provides a registration command.
4. **Reload/restart** the agent so the hook takes effect.

Once active, even if your session grows to hundreds of thousands of tokens, the core execution rules stay intact, preventing the agent from drifting into passive status‑updates mode.
